# 从初学者角度理解 Codex Agent 的四大模块

## 一、我为什么选择 Codex Agent

这次作业我选择分析 **Codex Agent**。原因是我现在正在使用它完成学习任务，所以它不是一个离我很远的概念，而是一个我可以直接观察的 AI 应用。

在刚接触 Agent 之前，我对 AI 的理解更接近“聊天机器人”：我提一个问题，它给我一个回答。但使用 Codex 之后，我发现 Agent 不只是会回答问题，它还可以理解任务背景、读取工作区文件、调用工具、执行命令，并且根据当前环境继续推进任务。

所以我理解的 Agent，不是单独的一个模型，而是由几个模块组合起来的系统。这里我用课程里提到的四个模块来分析 Codex Agent：`Context`、`Plugins`、`Model` 和 `Workspace`。

## 二、Context：Agent 如何理解任务和保持角色

### 1. Context 的作用

我理解的 `Context`，就是 Agent 当前能够看到和参考的信息。它包括用户输入、系统给它的规则、当前会话历史、环境信息，以及一些任务相关的约束。

对 Codex 来说，Context 很重要，因为它决定了 Agent 应该“以什么身份工作”和“按照什么规则工作”。如果没有 Context，模型可能只是在回答问题；但有了 Context，它就能更像一个有工作方式的助手。

### 2. System Prompt 像 Agent 的行为说明书

Codex 的 System Prompt 可以理解成一份“行为说明书”。它会告诉 Agent：

- 它是谁，比如一个偏软件工程方向的编码助手。
- 它应该怎么和用户沟通，比如直接、务实、清楚地说明做了什么。
- 它能做什么，比如读取文件、运行命令、修改代码。
- 它不能随便做什么，比如不能随意破坏用户已有文件，不能擅自执行危险操作。

我以前以为 AI 的回答主要取决于模型本身，但现在发现 System Prompt 对 Agent 的行为影响很大。它像是给模型加上了一层工作规范，让模型不只是“能说”，还知道“应该怎么做”。

### 3. 人格设定

Codex 有比较明显的人格设定。它不像娱乐型聊天助手那样追求轻松活泼，而是更像一个工程师：注重事实、注重代码质量、先观察环境再行动、尽量把事情完成。

这种人格设定的作用不是让 Agent 像人一样有情绪，而是让它在处理任务时形成稳定的风格。例如，当我让它完成报告时，它会先根据目标组织结构，而不是随便写一段泛泛的介绍。

### 4. 记忆机制

Codex 的记忆主要来自当前会话中的上下文。它能记住我前面说过“我刚刚接触 Agent”，所以后面写报告时就会调整成初学者视角。

但我认为这种记忆不等于人类的长期记忆。它更像是在当前任务窗口里临时保存信息。如果会话很长，系统可能会压缩前面的内容，用摘要继续保持任务连续性。真正长期保存的内容，更多还是依赖文件、Git、配置或外部工具。

### 5. 配置方式

Context 的配置通常不是普通用户直接写代码完成的，而是通过以下方式影响：

- 系统层面的 System Prompt。
- 用户在对话中输入的要求。
- 当前工作区提供的文件和环境信息。
- 项目里的说明文档、配置文件或技能说明。

所以我理解 Context 就像 Agent 的“任务背景”和“行为边界”。

## 三、Plugins：Agent 如何调用工具完成事情

### 1. Plugins 的作用

如果说 Model 是 Agent 的大脑，那么 `Plugins` 就像 Agent 的手和工具箱。模型本身主要负责理解和生成内容，但它不能直接操作现实环境。Plugins 让 Agent 可以真正做事。

例如，Codex 可以调用终端命令、读取文件、编辑文件、查看 Git 状态、搜索资料，甚至在某些场景下生成图片或使用外部连接器。

### 2. Codex 可以调用哪些工具

从我当前使用的 Codex 来看，它可以使用的工具包括：

- 文件系统工具：读取、搜索、修改工作区里的文件。
- 终端工具：运行 PowerShell 命令，查看项目结构或执行测试。
- Git 相关能力：查看分支、检查变更、提交代码等。
- 网络搜索工具：在需要最新信息时查询互联网。
- 自动化工具：设置提醒、周期任务或后续跟进。
- 图片生成工具：在需要视觉素材时生成图片。

这些工具让 Agent 不只是“回答我”，而是可以参与实际工作。

### 3. 技能包机制

Codex 还有技能包的概念。我的理解是，技能包就像某类任务的专门说明书。例如：

- `imagegen` 用于生成或编辑图片。
- `openai-docs` 用于查询 OpenAI 官方文档。
- `skill-creator` 用于创建新的技能。

技能包不是底层模型本身，而是给 Agent 增加了特定任务的流程和规范。这样 Agent 在遇到某类任务时，就可以按照更专业的步骤来处理。

### 4. 是否支持插件扩展

Codex 支持插件或连接器扩展，比如连接 GitHub、Gmail、Google Drive、Notion、Slack 等。这样 Agent 就可以从单一工作区扩展到更多外部系统。

我理解插件扩展的意义是：Agent 的能力边界不只由模型决定，还由它能连接哪些工具决定。一个没有插件的模型可能只能聊天；一个有插件的 Agent 可以读文件、查资料、改代码、整理任务，甚至和其他软件协作。

### 5. 配置方式

Plugins 的配置方式一般包括：

- 系统预先开放哪些工具。
- 用户是否授权某个插件或连接器。
- 技能包是否安装在本地。
- 当前任务是否允许调用对应工具。

所以 Plugins 可以看作 Agent 的“行动能力”。

## 四、Model：Agent 的底层推理能力

### 1. Model 的作用

`Model` 是 Agent 的核心推理部分，也就是我理解中的“大脑”。它负责理解我的问题、分析任务、生成文字、判断下一步该做什么。

在 Codex 中，底层模型属于 GPT 系列的 Codex 能力形态。它特别适合处理代码、文件、项目结构和工程任务。

### 2. 模型不等于完整 Agent

我一开始容易把 Model 和 Agent 混在一起，以为一个强大的模型就是 Agent。现在我觉得这两个概念不一样。

Model 更像是会思考和表达的大脑，但 Agent 是一个完整系统。Agent 除了 Model 之外，还需要 Context 来理解规则，需要 Plugins 来调用工具，需要 Workspace 来管理工作现场。

所以一个模型可以回答“怎么做”，但一个 Agent 可以在一定权限内真的去“做”。

### 3. 模型选择策略

Codex 可以根据任务复杂度选择不同的模型或推理强度。例如：

- 简单问答可以使用更快、更轻量的模型。
- 复杂代码分析可以使用推理能力更强的模型。
- 大型项目修改可能需要更高的推理强度。

这种策略说明 Agent 不一定每次都用同一种方式工作，而是会在速度、成本和质量之间做平衡。

### 4. 配置方式

Model 的配置通常包括：

- 选择具体模型版本。
- 设置推理强度。
- 设置是否允许调用工具。
- 通过 System Prompt 限制模型的行为方式。

我理解 Model 是 Agent 的“思考能力”，但它必须和其他模块配合，才能完成真实任务。

## 五、Workspace：Agent 如何管理工作现场和历史

### 1. Workspace 的作用

`Workspace` 可以理解成 Agent 做事的“现场”。对于 Codex 来说，Workspace 包括当前文件夹、项目代码、终端环境、Git 仓库、配置文件和任务相关资料。

比如我现在的工作区就是一个本地目录。Codex 可以查看这个目录里有什么文件，并根据这些文件决定下一步怎么处理。

### 2. 会话历史管理

Codex 会利用当前会话历史来保持连续性。比如前面我说“你需要站在我的角度去完成作业，我刚刚接触 agent”，后面它就应该在报告中使用初学者视角。

当会话内容变多时，系统可能不会完整保留每一句话，而是通过摘要或压缩后的上下文继续工作。这样可以在有限上下文窗口里保持任务方向。

### 3. 持久化存储

我理解的持久化，就是任务结束后仍然能保留下来的东西。Codex 的持久化不主要靠“记住我这个人”，而是靠外部载体，例如：

- 本地工作区中的文件。
- Git 仓库里的提交记录。
- 自动化任务配置。
- 插件连接的外部服务，比如云盘、邮箱、项目管理工具。

这说明 Workspace 不只是一个临时聊天窗口，而是 Agent 能持续工作的基础。

### 4. 配置方式

Workspace 的配置方式包括：

- 指定当前工作目录。
- 设置文件读写权限。
- 配置 Git 仓库。
- 配置终端环境和依赖。
- 连接外部存储或插件服务。

所以 Workspace 是 Agent 的“工作环境”和“状态保存位置”。

## 六、我的思考：Agent 不是一个聊天模型，而是一个被组织起来的工作系统

通过这次分析，我对 Agent 的理解发生了变化。

刚开始我以为 Agent 就是一个更聪明的聊天机器人。但现在我觉得，Agent 更像是一个由多个模块组成的工作系统：

- `Context` 决定它知道什么、遵守什么规则。
- `Plugins` 决定它能调用什么工具、能做什么动作。
- `Model` 决定它如何理解、推理和生成方案。
- `Workspace` 决定它在哪里工作、如何保存状态。

这四个模块组合起来，才让 Agent 具备“理解任务并执行任务”的能力。

我认为学习 Agent 时，不能只关注模型有多强，也要关注它的上下文设计、工具权限、工作区管理和安全边界。因为在真实使用中，Agent 的能力不是单纯来自模型，而是来自整个系统的协作。

对我这个刚接触 Agent 的学习者来说，最重要的收获是：Agent 的核心不是“会聊天”，而是“在一定规则和工具支持下，能持续完成任务”。
