## 二、Context：Agent 如何理解任务和保持角色

### 1. Context 的作用

我理解的 `Context`，就是 Agent 当前能够看到和参考的信息。它包括用户输入、系统给它的规则、当前会话历史、环境信息，以及一些任务相关的约束。

它决定了 Agent 应该“以什么身份工作”和“按照什么规则工作”。如果没有 Context，模型可能只是在回答问题；但有了 Context，它就能更像一个有工作方式的助手。
context包括当前会话上下文（short-term context），工作记忆（working memory），长期记忆（long-term memory），system prompt.md

### 2. System Prompt 像 Agent 的行为说明书

Codex 的 System Prompt 可以理解成一份“行为说明书”。在用户消息之前，给模型的一层“最高优先级指令”，它会告诉 Agent：

- 它是谁，比如一个偏软件工程方向的编码助手。
- 它应该怎么和用户沟通，比如直接、务实、清楚地说明做了什么。
- 它能做什么，比如读取文件、运行命令、修改代码。
- 它不能随便做什么，比如不能随意破坏用户已有文件，不能擅自执行危险操作。

### 4. 记忆机制

分几种，（1）短期记忆 Short-term Memory

通常：

当前任务状态

（2）长期记忆 Long-term Memory

## 三、Plugins：Agent 如何调用工具完成事情

### 1. Plugins 的作用

模型本身主要负责理解和生成内容，但它不能直接操作现实环境。Plugins 让 Agent 可以真正做事。


### 2. Codex 可以调用哪些工具

可以使用的工具包括：

- 文件系统工具：读取、搜索、修改工作区里的文件。
- 终端工具：运行 PowerShell 命令，查看项目结构或执行测试。
- Git 相关能力：查看分支、检查变更、提交代码等。
- 网络搜索工具：在需要最新信息时查询互联网。
- 自动化工具：设置提醒、周期任务或后续跟进。
- 图片生成工具：在需要视觉素材时生成图片。

这些工具让 Agent 不只是“回答我”，而是可以参与实际工作。

### 3. 是否支持插件扩展

Codex 支持插件或连接器扩展，比如连接 GitHub、Gmail、Google Drive、Notion、Slack 等。这样 Agent 就可以从单一工作区扩展到更多外部系统。

Agent 的能力边界不只由模型决定，还由它能连接哪些工具决定。一个没有插件的模型只能聊天；一个有插件的 Agent 可以读文件、查资料、改代码、整理任务，甚至和其他软件协作。



## 四、Model：Agent 的底层推理能力

### 1. Model 的作用

`Model` 是 Agent 的核心推理部分，负责理解我的问题、分析任务、生成文字、判断下一步该做什么。

在 Codex 中，底层模型属是GPT 的各种系列。泛化能力强，综合能力强

### 2. 模型不等于完整 Agent


Model 会思考和表达的大脑，但 Agent 是一个完整系统。Agent 除了 Model 之外，还需要 Context 来理解规则，需要 Plugins 来调用工具，需要 Workspace 来选择，管理工作现场。

所以一个模型可以回答“怎么做”，但一个 Agent 可以在一定权限内真的去“做”。

### 3. 模型选择策略

根据任务复杂度设置推理强度，如low，medium，high，extra，high
也可以选择模型版本5.5,5.4等
也可以选择牺牲token用量从standard标准切换到fast（1.5倍速度，2.5倍用量）



## 五、Workspace：Agent 如何管理工作现场和历史

### 1. Workspace 的作用

`Workspace` 可以理解成 Agent 做事的“现场”。对于 Codex 来说，Workspace 包括当前文件夹、项目代码、终端环境、Git 仓库、配置文件和任务相关资料。

比如我现在的工作区就是一个本地目录。Codex 可以查看这个目录里有什么文件，并根据这些文件决定下一步怎么处理。

### 2. 会话历史管理

Codex 会利用当前会话历史来保持连续性。比如前面我说“你需要站在我的角度去完成作业，我刚刚接触 agent”，后面它就应该在报告中使用初学者视角。

当会话内容变多时，系统可能不会完整保留每一句话，而是通过摘要或压缩后的上下文继续工作。这样可以在有限上下文窗口里保持任务方向。

### 3. 持久化存储

持久化存储是任务结束后仍然能保留下来的东西。Codex 的持久化靠外部载体，例如：

- 本地工作区中的文件。
- Git 仓库里的提交记录。
- 自动化任务配置。
- 插件连接的外部服务，比如云盘、邮箱、项目管理工具。

这说明 Workspace 不只是一个临时聊天窗口，而是 Agent 能持续工作的基础。

### 4. 配置方式

Workspace 的配置方式包括：

- 指定当前工作目录。
- 设置文件读写权限。
- 配置 Git 仓库。
- 配置终端环境和依赖。
- 连接外部存储或插件服务。

所以 Workspace 是 Agent 的“工作环境”和“状态保存位置”。
